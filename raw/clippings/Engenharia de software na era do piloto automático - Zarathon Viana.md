## Processo seletivo na era da IA

**Matos:** Uma coisa que tem tomado a parte do que eu tenho pensado ultimamente é sobre o processo seletivo. Antigamente a gente mandava um trabalho para algum projeto pra pessoa fazer em casa ou então ela participava de uma entrevista de algoritmo ou system design. Mais recentemente, quando a gente mandava um desafio, assim, magicamente todo mundo mandava a resolução. O que aconteceu? Basicamente pessoas estavam usando IA para resolver. Então se a gente queria ver como que a pessoa resolve o projeto, talvez um desafio para casa já não faça mais sentido. E tem outras coisas: será que a pessoa hoje precisa ser muito boa em algoritmo?

**Zara:** Ele muda, eu acho que ele muda. Assim como a gente não pede mais para ninguém escovar bit no processo seletivo, que provavelmente acontecia no passado, agora a gente não vai precisar menos ainda. E eu acho que a gente começa agora muito mais a identificar padrões criativos de resolver problemas.

E eu acho que o processo muda muito mais para uma linha mais prática, tipo, tem um problema, o cara vai abrir um codebase cagado e vai dizer: "Ó, tá aqui. Como é que você resolve esse problema?" Ou tipo "esse software tem um problema, descubra quais os problemas que esse software tem" e aí você vai deixar na mão do candidato cavucar de uma maneira criativa e você vai entender o quão criativo esse cara é, por onde ele foi, se ele seguiu um padrão habitual, se ele foi fora da casinha.

De novo, se você sabe os conceitos, se você tem uma base sólida, utilizar IA para investigar um problema é dois palitos. Agora, se você não tem, putz, você vai rodar muito e vai ficar claro quem tem uma base sólida e quem não tem.

Cada vez mais, e aí a gente vai falar de LLMs, LLM é muito sobre comunicação, né? Sobre pedido. Eu peço algo e eu recebo algo de volta. Então, a habilidade de se expressar vai ser cada vez mais exigida. A habilidade de se comunicar, ser direto, tirar ambiguidades da fala. A coisa do harness, né? Como é que você cria um harness na comunicação também? Harness não é só tooling, harness também é comunicação. Isso também vai ter que ser exigido. Agora, tipo, como é que esse cara é token efficient? Porque token é caro. Então, se esse cara não consegue se comunicar bem, ser direto, ele pode perder muito tempo nesse pingpong.

Quanto mais perto ou quanto mais próximo de você declarar sua intenção legitimamente, diretamente, pragmaticamente, melhor. Se você for muito ambíguo, cara, a LLM vai para um canto, você vai ter que esperar o negócio rodar para depois dizer: "Putz, não era isso, volta" e aí o contexto tá poluído e azedou. Então tem que entender muito agora sobre o jogo da comunicação, como é que você se comunica com a máquina.

## Sanity checks e código que envelhece

**Matos:** Eu já vi muitos times tendo dificuldade tomando decisões muito ruins ao longo da vida de um projeto. Decisões ruins acabam se acumulando até levar o projeto para um ponto ruim. Agora existe a possibilidade de a pessoa criar código ruim porque a IA fez output, funcionou, ela aceita e vai embora sem revisar. Num caso desse, como é que faz para a gente descobrir mais cedo que a gente tá tendo esse tipo de problema?

**Zara:** Eu chamo isso de sanity check, tá? É um cheque de sanidade do teu codebase, o quanto meu código tá sadio. Eu acho que isso precisa estar dentro de um CI, assim, dentro de um negócio automatizado, que antes de ir pro ar, ele faz um cheque daquela mudança, daquele código novo e diz: "Beleza, isso tá compliant, isso ainda mantém o código sadio." Porque de novo eu vou usar a IA contra a IA. E posso brincar de ter multi-julgamentos de multimodelos para saber se a interseção de todos esses outputs é o mesmo. Aí tem várias formas de fazer isso: multimodelos, multiagentes.

Mas eu acho que você precisa de um sanity check antes do codebase. Você pode antecipar essa camada lá pra camada de dev ainda, sem mexer no código. Lembra? A gente usava o Sonar, né? A gente velha de guerra usava o Sonar para fazer inspeção de código estático. A gente vai fazer a mesma coisa agora, só que pro codebase inteiro, porque agora é mais barato fazer isso. E aí você pode ter esses sanity checks. No fim do dia são arreios, né? São os harnesses antes do código ir para produção.

A proposta é que eu faço isso de certa forma não intencional quando eu, Zara, estou avaliando a proposta de solução no nível de design da solução que Opus me deu. Eu faço esse sanity check: "Putz, mas acho que isso não envelhece bem." E aí quando eu tenho dúvida pergunto: "Ó, isso envelhece bem nesse contexto ou envelhece mal? Quais são os tradeoffs?" E isso acaba indo também pra minha ADR quando eu escrevo. Na ADR diz: "A gente tá abrindo mão disso, a gente tá comprando isso, a gente tem ganho aqui." Se você faz isso constantemente, você tem um código que tá legal.

Se você negligencia isso aí, paga o preço daqui a pouco. Então assim, você escolhe: queima mais token pelo menos você não tem que voltar nesse negócio daqui a três, quatro semestres, entendeu? Porque código azeda, código azeda. Isso é fato. O código de hoje resolve o problema de hoje. O código de amanhã, que provavelmente é um filho do código de hoje, vai levar todos os pecados do pai dele para lá.

**Matos:** Existe até o conceito de fitness functions, né? Talvez menos usado no mercado que deveria, mas agora vai ficar mais simples. Checagem determinística quando dá para usar, vamos usar. Em alguns casos, talvez o determinístico não funcione tão bem, então a gente pode fazer algo usando LLM. Cada empresa vai ter que ter as suas convenções de arquitetura ou code guidelines. E isso vai precisar estar dentro desse circuito de cheques, tanto para construir quanto para avaliar no final.

**Zara:** A parte mais importante é a avaliação do final. Pessoal pensa: "Ah, engenharia de contexto é legal", é, mas ela é o começo da conversa. O mais importante é o código que tá ainda na camada de dev, ele é mais barato de ser corrigido. Depois que vai pro ar, cara, quanto mais perto da produção, mais custoso é esse software do ponto de vista de manutenção. Quanto antes você conseguir antecipar, melhor.

O Martin Fowler, né, o feed forward é antes, mas eu acho que a camada de feedback é mais importante ainda: uma vez que esse código foi produzido, como é que eu garanto que isso tá compliant com o que a gente definiu no começo? Isso tá batendo com minha code architecture? Tá compliant? A gente precisa criar documentação. Cara, documentação ficou muito barata. Documentação de código hoje basicamente é muito barato criar, manter, evoluir. E manter isso ficou muito mais fácil. Você bota no CI, o CI gera o código, gera doc, atualiza doc, faz tudo.

É, ficou tão simples fazer documentação ultimamente que talvez até a gente tenha que tomar cuidado para não documentar demais, além do necessário.
